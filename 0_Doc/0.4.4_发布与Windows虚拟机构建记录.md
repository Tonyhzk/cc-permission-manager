## 本次产物

- macOS：`release/0.4.4/CC Permission Manager_0.4.4_universal.dmg`
- Windows NSIS：`release/0.4.4/CC Permission Manager_0.4.4_x64-setup.exe`

## 这次踩到的坑

### 1. Mac 侧一开始访问不到 Windows 文件

现象：

- 之前能直接访问 Windows 文件
- 迁移电脑后，`/Volumes/[C] Windows 11/` 一开始不存在
- Windows 虚拟机里却能正常访问 Mac 路径

排查结果：

- Parallels 里的 `UseInversedDisks` 需要开启
- 实际挂载不是彻底没了，而是先以隐藏路径挂到 `/Volumes/.PEVolumes/`
- 后来恢复成了可见路径：`/Volumes/[C] Windows 11`
- 当前实际挂载方式是 **SMB**，不是 macFUSE

验证命令：

```bash
mount | grep "[C] Windows 11"
```

可见结果里是：

```bash
//GUEST:@Windows 11._smb._tcp.local/[C] Windows 11 on /Volumes/[C] Windows 11 (smbfs ...)
```

说明当前是 SMB 挂载。

### 2. macFUSE 装了，但这次最终不是靠它生效

排查时确认了：

- `macfuse.fs` 已安装
- 但系统扩展列表里没有看到 macFUSE 正在工作
- 最后恢复访问能力的还是 Parallels 的 SMB 共享路径

结论：

- 这次“能访问 Windows 文件”这件事，最终不是 macFUSE 直接接管
- 真正恢复构建流程的关键，是找到可写目录并用它做同步和构建

### 3. 不是整个 Windows 盘都可写

一开始最容易误判成“整个盘只读”。

实际测试结果：

**可写目录**：

- `/Volumes/[C] Windows 11/Users/jjp`
- `/Volumes/[C] Windows 11/Users/jjp/Documents`
- `/Volumes/[C] Windows 11/Users/jjp/Documents/GIthub`
- `/Volumes/[C] Windows 11/Users/Public`
- `/Volumes/[C] Windows 11/temp`

**不可写目录**：

- `/Volumes/[C] Windows 11/`
- `/Volumes/[C] Windows 11/Users`
- `/Volumes/[C] Windows 11/Users/Windows/Documents`

最终采用的 Windows 项目目录：

```bash
/Volumes/[C] Windows 11/Users/jjp/Documents/GIthub/cc-permission-manager
```

## Windows 虚拟机构建方式

虚拟机：

- Parallels VM：`Windows 11`

核心方式：

- 在 macOS 上通过 `prlctl exec` 远程执行 Windows 命令
- 在 Windows 内使用现有脚本和本地编译环境完成构建

### 基础检查

检查过的关键环境：

- `pnpm`
- `cargo`
- Visual Studio 2022 Build Tools

### 第一个构建失败点：脚本跑在了错误用户环境

`win_build_x64.bat` 默认拿的是：

```bash
C:\Windows\syswow64\config\systemprofile\.cargo\bin
```

所以第一次直接报：

```bash
[ERROR] Cargo not found at: C:\Windows\syswow64\config\systemprofile\.cargo\bin
```

原因：

- `prlctl exec` 执行批处理时，环境并不是 `jjp` 的用户环境
- Rust 和 pnpm 实际装在 `C:\Users\jjp` 下面

解决方法：

构建时显式指定这些环境变量：

```bash
USERPROFILE=C:\Users\jjp
HOMEDRIVE=C:
HOMEPATH=\Users\jjp
APPDATA=C:\Users\jjp\AppData\Roaming
LOCALAPPDATA=C:\Users\jjp\AppData\Local
CARGO_HOME=C:\Users\jjp\.cargo
RUSTUP_HOME=C:\Users\jjp\.rustup
PATH=C:\Users\jjp\.cargo\bin;C:\Users\jjp\AppData\Roaming\npm;%PATH%
```

### 第二个构建失败点：Tauri 版本不匹配

Windows 里第一次 `pnpm install` 后，解析到了更高版本依赖，导致：

- `tauri` Rust crate：`2.9.5`
- `@tauri-apps/api`：`2.10.1`
- `@tauri-apps/plugin-dialog`：`2.7.0`

然后 `tauri build` 报版本不匹配。

原因：

- 项目 `package.json` 里用了 `^`
- Windows 端重新安装依赖时，拿到了更新的小版本
- Rust 侧和 JS 侧没有一起升级

解决方法：

在 **Windows 构建副本** 里把版本固定成和项目当前可用组合一致，然后删除 `node_modules` 重新安装。

最终固定为：

- `@tauri-apps/cli`: `2.8.0`
- `@tauri-apps/api`: `2.8.0`
- `@tauri-apps/plugin-dialog`: `2.4.0`
- `@tauri-apps/plugin-os`: `2.0.0`
- `@tauri-apps/plugin-shell`: `2.0.0`
- `@tauri-apps/plugin-store`: `2.0.0`

## 最终成功构建命令思路

构建过程本质上是：

1. 切到 Windows 项目目录下的 `src`
2. 显式切到 `jjp` 的 Rust / pnpm 环境
3. 调用 VS Build Tools 的交叉编译环境
4. 执行 Tauri x64 构建

关键命令结构：

```bash
call "C:\Program Files (x86)\Microsoft Visual Studio\2022\BuildTools\VC\Auxiliary\Build\vcvarsall.bat" arm64_amd64
pnpm tauri build --target x86_64-pc-windows-msvc
```

这里的 `arm64_amd64` 很关键，表示在 ARM64 Windows 主机里交叉编译 x64 安装包。

## 下次再做时的最短路线

### macOS 包

直接运行：

```bash
bash 2_Scripts/mac_build.sh
```

选择 `3`，构建 universal。

### Windows 包

建议按下面顺序：

1. 先确认可写路径还是：
   ` /Volumes/[C] Windows 11/Users/jjp/Documents/GIthub `
2. 同步 `src/`，忽略 `node_modules / dist / src-tauri/target`
3. 按需补齐根目录文件
4. 确认 Windows 构建副本里的 Tauri 依赖版本没有漂移
5. 用 `jjp` 环境变量执行构建
6. 构建完成后把 `exe / msi` 拷回 `release/<版本号>/`

## 本次最重要的结论

1. **不是虚拟机完全访问不到，而是要先分清“可见”“可写”“真实挂载方式”这三件事**
2. **Windows 可写目录要用 `Users/jjp/Documents/GIthub`，不要直接往盘根目录写**
3. **`prlctl exec` 跑批处理时，默认用户环境不可靠，必须显式指定 Rust / pnpm 路径**
4. **Tauri 依赖一旦重新安装，JS 侧小版本可能漂移，必须和 Rust 侧保持一致**
